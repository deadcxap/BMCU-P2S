# Структура репозитория BMCU-P2S (обзор на русском)

## 1) Общая архитектура
Проект — это прошивка для контроллера BMCU на базе CH32V203 (RISC‑V), собираемая через PlatformIO.
Логически код делится на:

- **Точка входа и orchestration**: `src/main.cpp`
- **Логика протяжки и состояний филамента**: `src/Motion_control.cpp`, `src/MC_PULL_calibration.cpp`
- **Протоколы обмена с принтером/шиной AMS**: `src/bambu_bus_ams.cpp`, `src/ahub_bus.cpp`, `src/_bus_hardware.cpp`
- **Работа с датчиками и периферией**: `src/ADC_DMA.cpp`, `src/many_soft_AS5600.cpp`, `src/ws2812.cpp`
- **Постоянное хранение параметров (Flash)**: `src/Flash_saves.cpp`
- **Служебные модули**: `src/Debug_log.cpp`, `src/hal/time_hw.c`, `src/crc_bus.c`

## 2) Назначение ключевых файлов

### Конфигурация сборки
- `platformio.ini`
  - Определяет платформу/плату и compile-time флаги.
  - В `env:fw` берёт параметры из переменных окружения (`BAMBU_BUS_AMS_NUM`, `AMS_RETRACT_LEN`, и т.д.), что позволяет массово собирать варианты прошивки.

### Точка входа и главный цикл
- `src/main.cpp`
  - Инициализирует MCU, таймеры, RGB, АЦП, флеш-память, шины.
  - Загружает сохранённые параметры и состояние канала.
  - В бесконечном цикле:
    1. Обрабатывает пакеты AHUB/BambuBus.
    2. Запускает управление движением.
    3. Обновляет индикацию RGB.
    4. Периодически сохраняет состояние в Flash.

### Управление механикой и состояниями
- `src/Motion_control.cpp` / `src/Motion_control.h`
  - Основная машина состояний подачи/втягивания филамента.
  - Логика DM autoload (двухкнопочный/двухдатчиковый сценарий).
  - Ограничения, целевые скорости, PID/ШИМ, защитные механизмы.
- `src/MC_PULL_calibration.cpp` / `src/MC_PULL_calibration.h`
  - Калибровка каналов и связанные параметры.

### Коммуникация
- `src/_bus_hardware.cpp` / `src/_bus_hardware.h`
  - Низкоуровневый UART+DMA драйвер обмена.
  - Буферизация, переключение направления RX/TX, обработка кадров.
- `src/bambu_bus_ams.cpp` / `src/bambu_bus_ams.h`
  - Реализация протокола AMS со стороны BMCU.
  - Сборка/разбор кадров, CRC, реакции на команды/heartbeat.
- `src/ahub_bus.cpp` / `src/ahub_bus.h`
  - Логика второй шины (AHUB), маршрутизация пакетов.
- `src/crc_bus.c` / `src/crc_bus.h`
  - CRC8/CRC16 для протоколов.

### Датчики и периферия
- `src/ADC_DMA.cpp` / `src/ADC_DMA.h`
  - Снятие аналоговых значений через DMA.
- `src/many_soft_AS5600.cpp` / `src/many_soft_AS5600.h`
  - Программная многоканальная работа с AS5600.
- `src/ws2812.cpp` / `src/ws2812.h`
  - Управление WS2812 (включая тайминги и профили цветов).

### Хранение и отладка
- `src/Flash_saves.cpp` / `src/Flash_saves.h`
  - Чтение/запись конфигурации, калибровок и состояния в Flash.
- `src/Debug_log.cpp` / `src/Debug_log.h`
  - Отладочный UART-лог.
- `src/hal/time_hw.c` / `src/hal/time_hw.h`
  - Низкоуровневая тайм-база.

## 3) Взаимосвязи модулей (кратко)

1. `main.cpp` вызывает `Motion_control_run()` и обработчики шин `ahubus_run()/bambubus_run()`.
2. `bambu_bus_ams.cpp` обновляет состояние структуры `ams`, которое используется в `Motion_control.cpp`.
3. `Motion_control.cpp` читает сенсоры (через `ADC_DMA` / `many_soft_AS5600`) и управляет моторами.
4. `Flash_saves` служит персистентным слоем для:
   - данных катушек,
   - калибровок,
   - состояния загруженного канала.
5. `ws2812` отображает текущее состояние устройства (онлайн/ошибка/активность).

## 4) Поток выполнения при старте

1. Инициализация MCU и периферии.
2. Инициализация модулей (лог, AMS, Flash, ADC).
3. Загрузка калибровок и данных катушек из Flash.
4. Восстановление состояния ранее загруженного канала.
5. Переход в рабочий цикл протоколов + motion control.

## 5) Как вносить изменения безопасно

- Изменения в логике подачи (`Motion_control`) всегда проверять на:
  - idle,
  - load / hold,
  - on_use,
  - pullback,
  - обработку ошибок/таймаутов.
- Изменения протокола (`bambu_bus_ams`) проверять вместе с CRC и длинами кадров.
- Изменения структуры сохраняемых данных (`Flash_saves`) делать с учётом backward-совместимости (magic/version).
- Перед релизом собирать несколько вариантов конфигурации (A1/P1S, autoload, RGB, AMS slot).

## 6) Новый CI-метод сборки

Вместо локального shell-скрипта используется GitHub Actions workflow:

- `.github/workflows/build-all-firmwares.yml`
- `scripts/ci/build_all_firmwares.py`

Workflow запускается вручную (`workflow_dispatch`) или при пуше в `main/master`, собирает все комбинации параметров и публикует архив результатов как artifact.
